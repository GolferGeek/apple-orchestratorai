# Workflow JSON

Workflow JSON files define structure, not live run state.

They should include:

- workflow identity
- lifecycle status: `dev`, `test`, `prod`, `archived`
- user-facing purpose and explanation metadata
- graph/subgraph/work-unit structure
- required Hermes skills
- required MCP resources/connectors
- expected inputs
- expected outputs
- human-in-the-loop checkpoints
- render hints

Live run state belongs in the local observability database.

## Explanation Metadata

Workflow JSON should include enough explanation metadata for Hermes to help users understand and refine the workflow.

Recommended fields:

- `purpose`
- `when_to_use`
- `user_level_summary`
- `assumptions`
- `inputs_summary`
- `outputs_summary`
- `human_review_summary`
- `safe_refinements`

Graphs, subgraphs, work units, and human tasks may also include explanation fields:

```json
{
  "id": "extract_and_classify",
  "type": "subgraph",
  "title": "Extract and Classify Documents",
  "purpose": "Turn selected files into a structured document inventory.",
  "explanation": "This step reads files, extracts text, identifies document type, and prepares the material for later summary and review steps.",
  "safe_refinements": [
    "change document categories",
    "add a human review checkpoint",
    "require citations for each classification"
  ]
}
```

The app should normally render Hermes-generated explanations, not the raw JSON.

## Minimal Shape

```json
{
  "schema_version": "0.1",
  "workflow_id": "wf_matter_intake",
  "name": "Matter Intake",
  "status": "dev",
  "purpose": "Help a legal user understand a new matter quickly.",
  "when_to_use": "Use after collecting the first set of matter files and optional client/matter data.",
  "required_skills": ["legal.collect_context", "legal.intake_summary"],
  "resources": [],
  "graph": {
    "nodes": [],
    "edges": []
  },
  "outputs": []
}
```

## Final Output Contract

Every workflow should declare a final output contract, even if the exact file contents are generated by Hermes skills.

The final output may be:

- a single document
- multiple files
- a structured JSON result
- a Markdown report
- a PDF
- a DOCX
- a table/spreadsheet
- an artifact bundle
- a folder/package containing several outputs

Example:

```json
{
  "outputs": [
    {
      "id": "final_package",
      "label": "Final Matter Intake Package",
      "type": "artifact_bundle",
      "required": true,
      "items": [
        {
          "id": "intake_summary",
          "label": "Intake Summary",
          "format": "markdown",
          "render_as": "document"
        },
        {
          "id": "fact_table",
          "label": "Fact Table",
          "format": "json",
          "render_as": "table"
        },
        {
          "id": "source_citations",
          "label": "Source Citations",
          "format": "json",
          "render_as": "list"
        }
      ]
    }
  ]
}
```

The contract should define what the workflow promises to produce, not necessarily how each item is generated.

## Graph-Level Output Contracts

Output contracts are not only workflow-level. A graph, subgraph, or work unit may also declare an output contract.

This matters especially when a graph pauses for human review. If a human is asked to approve, reject, or request changes, the app needs to know what is being reviewed.

Examples:

- a graph produces a draft package for approval
- a subgraph produces extracted facts for correction
- a work unit produces a table for review
- a human checkpoint presents a proposed final output with approve/change actions

Example:

```json
{
  "graph": {
    "nodes": [
      {
        "id": "human_review",
        "type": "human_task",
        "title": "Review extracted facts",
        "review_subject": "fact_extraction_package",
        "actions": ["approve", "request_changes", "reject"],
        "outputs": [
          {
            "id": "fact_extraction_package",
            "type": "artifact_bundle",
            "render_as": "review_package",
            "items": [
              { "id": "fact_table", "format": "json", "render_as": "table" },
              { "id": "source_citations", "format": "json", "render_as": "list" }
            ]
          }
        ]
      }
    ]
  }
}
```

The UI can then render the review package generically without knowing the workflow.

## Segmented Human Review

Human review may be segmented. Some workflows need the user to approve, reject, or modify multiple parts of an output package rather than one whole artifact.

Examples:

- approve/reject/modify each contract clause issue
- approve selected hot documents in discovery review
- approve individual timeline events
- approve extracted facts one by one
- approve memo sections separately
- approve generated files before final package assembly

A human task should be able to declare review segments:

```json
{
  "id": "clause_review",
  "type": "human_task",
  "title": "Review contract issues",
  "review_mode": "segmented",
  "actions": ["approve_all", "submit_segment_decisions", "request_global_changes"],
  "segments": [
    {
      "segment_id": "issue_001",
      "label": "Limitation of Liability",
      "output_ref": "contract_issue_001",
      "allowed_decisions": ["approve", "modify", "reject"]
    },
    {
      "segment_id": "issue_002",
      "label": "Termination",
      "output_ref": "contract_issue_002",
      "allowed_decisions": ["approve", "modify", "reject"]
    }
  ]
}
```

The runtime should store each segment decision, not only the top-level human task result.

Workflows may also include multiple human checkpoints. Each checkpoint should identify:

- what is being reviewed
- whether review is whole-package or segmented
- allowed actions
- what happens after approval
- what happens after requested changes
- what happens after rejection
